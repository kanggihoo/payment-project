# Phase 1: Naive Payment 회고 및 인사이트

## 💡 여기서 얻은 인사이트

- **외부 API를 `@Transactional` 안에서 호출하면 커넥션 풀이 고갈된다**
  - PG 응답 10초 × 풀 크기 10 = 10개 커넥션이 10초간 점유 → 11번째 요청부터 대기/실패
  - 이로 인해 결제 기능뿐만 아니라 DB를 사용하는 시스템의 다른 기능(로그인, 조회 등)까지 마비될 수 있다.
- **PG 승인 성공 후 DB 저장에서 예외가 발생하면 "돈은 빠졌는데 기록은 없는" 상태가 된다**
  - 트랜잭션의 원자성(Atomicity)이 외부 시스템까지는 보장되지 않기 때문이다.
- **이것이 "외부 시스템과의 분산 트랜잭션"이 불가능한 근본적 이유**
  - 롤백할 수 없는 외부 액션(결제)을 트랜잭션에 포함시키는 설계는 매우 위험하다.

---

## 📝 회고

- **예상과 달랐던 점**: 
  - 단순히 "느려진다"고만 생각했는데, 실제 테스트를 통해 커넥션 풀이 순식간에 고갈되어 시스템이 거부되는 현상을 숫자로 확인하니 그 위험성이 훨씬 피부로 와닿았다.
- **가장 어려웠던 점**:
  - 멀티스레드 환경에서 풀 고갈 현상을 정확하게 재현하기 위해 `AtomicInteger`, `ExecutorService`, `WireMock` 지연 시간을 정밀하게 맞추는 부분이 까다로웠다.
- **실무에 적용한다면**:
  - 외부 API 호출은 반드시 트랜잭션 외부로 분리하고, 타임아웃 설정을 보수적으로 가져가야겠다.
  - 데이터 불일치에 대비한 보상 트랜잭션이나 로깅 전략이 필수적임을 깨달았다.
- **핵심 수치 요약**: 
  - 커넥션 풀(5개) 고갈까지 **약 3초**(Hikari Timeout), 불일치 발생 **100%**(Spy 테스트 기준)
