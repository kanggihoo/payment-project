# Memory

이 디렉터리는 AI와 장기 프로젝트를 이어서 진행하기 위한 메모리 시스템이다.
목적은 모든 기록을 보관하는 것이 아니라, context window가 초기화된 AI가 현재 상태와 작업 기준을 빠르게 복원하도록 돕는 것이다.

## 읽는 순서

세션 시작 시 이 파일을 먼저 읽은 뒤, 아래 순서로 읽는다.

1. `project-brief.md`
2. `current-state.md`
3. `tasks.md`
4. `decisions.md`
5. `conventions.md`

필요한 경우에만 추가로 읽는다.

- `architecture.md`: 프로젝트 구조나 모듈 경계를 파악해야 할 때
- `known-issues.md`: 미해결 버그, 기술 부채, 위험 요소를 확인해야 할 때
- `archived/`: 오래된 결정이나 과거 맥락 확인이 필요할 때

문제가 발생했을 때만 검색해서 참고한다.

- `troubleshooting.md`: 반복되는 오류, 환경 문제, 해결 절차

## 각 파일 역할

- `readme.md`: 메모리 시스템의 사용 규칙과 운영 원칙
- `project-brief.md`: 프로젝트의 장기 목표, 핵심 사용자/사용 맥락, 제품적 경계, 장기 우선순위
- `current-state.md`: 현재 프로젝트 단계, 최근 작업 맥락, 바로 이어서 볼 파일, 주의할 점
- `tasks.md`: 다음에 실행할 작업 큐, 진행 중인 작업, 보류 작업
- `decisions.md`: 반복하면 안 되는 설계, 테스트, 운영 결정과 그 이유
- `conventions.md`: AI가 따라야 하는 작업 규칙, 코딩 스타일, 테스트 방식, Tech Stack
- `architecture.md`: 주요 모듈 책임, 데이터 흐름, 외부 의존성, 건드리면 위험한 경계
- `known-issues.md`: 아직 해결하지 않은 버그, 기술 부채, 임시 처리, 위험 요소
- `troubleshooting.md`: 자주 발생하는 문제, 원인, 확인 명령, 해결 절차
- `archived/`: 현재 판단에는 직접 쓰지 않는 오래된 메모리

## 갱신 주체

- 메모리 파일의 편집 작업은 기본적으로 AI가 수행한다.
- 사용자는 직접 파일을 수정하기보다 AI에게 갱신, 정리, 이동, 삭제를 요청한다.
- AI는 세션 진행에 필요한 `current-state.md`와 `tasks.md`를 스스로 갱신할 수 있다.
- AI는 사용자의 지시, 작업 결과, 발견한 불일치를 바탕으로 필요한 메모리를 갱신할 수 있다.
- `project-brief.md`, `conventions.md`, Tech Stack, 핵심 설계 원칙은 사용자의 명시 지시 또는 확인 후 AI가 수정한다.
- 사용자가 직접 편집한 내용은 우선 존중하며, AI는 임의로 되돌리지 않는다.

## 갱신 조건

### 세션 시작

- `.memory/readme.md`를 먼저 읽고, 지정된 순서대로 필요한 메모리를 읽는다.
- 세션 시작 시에는 메모리를 우선 신뢰하고, 전체 source code를 탐색하지 않는다.
- 세션 시작 시 테스트를 실행하지 않는다.
- source code, test, git 확인은 사용자의 요청, 작업 대상, 또는 메모리 이상 징후가 있을 때 필요한 범위에서만 수행한다.

### 세션 중

- 사용자가 설계 방향을 명시적으로 결정했거나 작업 중 합의된 설계 판단이 생기면 `decisions.md`에 기록한다.
- 작업 규칙이나 Tech Stack 변경이 필요하면 사용자 확인 후 `conventions.md`를 갱신한다.
- 구조적 변경이 생겨 기존 설명이 틀려졌다면 `architecture.md`를 갱신한다.
- 새로 발견한 위험이 당장 해결되지 않으면 `known-issues.md` 또는 `tasks.md`에 남긴다.
- 반복되는 오류와 해결 절차가 검증되면 `troubleshooting.md`에 남긴다.
- 작업 중 메모리와 코드의 불일치, 오래된 정보, 모순을 발견하면 사용자에게 알리고 필요한 갱신을 제안하거나 수행한다.

### 세션 종료

- `current-state.md`에는 다음 AI가 바로 이어받는 데 필요한 현재 상태만 짧게 남긴다.
- `current-state.md`는 30초 안에 읽고 현재 상황을 복원할 수 있는 분량을 유지한다.
- `tasks.md`에는 다음 행동, 진행 중인 일, 막힌 일을 갱신한다.
- 완료된 구현 세부 설명은 길게 남기지 않는다. 완료 이력은 source code와 git history를 신뢰한다.

## source code 확인 범위

세션 시작 시에는 메모리를 우선 읽고, 전체 source code를 탐색하지 않는다.

source code는 다음 경우에만 확인한다.

- 사용자가 특정 구현, 수정, 버그 조사를 요청한 경우
- `current-state.md` 또는 `tasks.md`에 확인할 파일이 명시된 경우
- 메모리만으로 다음 행동을 결정할 수 없는 경우
- 작업 중 메모리와 코드의 불일치가 의심되는 경우

확인 순서는 다음과 같다.

1. 메모리에 명시된 파일
2. 사용자가 언급한 파일
3. 파일명, 기능명, symbol 검색으로 찾은 후보 파일
4. 후보 파일과 직접 연결된 import/use 관계 파일

전체 source code 탐색은 기본 동작이 아니다.
필요하면 사용자에게 이유를 설명하고 진행한다.

## test 실행 범위

세션 시작 시 테스트를 실행하지 않는다.

테스트는 다음 경우에만 실행한다.

- 구현 또는 수정이 끝난 뒤 검증이 필요한 경우
- 사용자가 테스트 실행을 요청한 경우
- 기존 실패를 재현하거나 원인을 확인해야 하는 경우

메모리 시스템은 테스트 통과 여부를 보장하지 않는다.

## decisions.md 항목 포맷

`decisions.md`의 각 결정은 아래 포맷을 따른다.

```md
## 결정 제목

- 날짜: YYYY-MM-DD
- 상태: active | superseded | archived
- 결정: ...
- 이유: ...
- 버린 대안: ...
- 재검토 조건: ...
```

작성 기준:

- `결정`에는 최종 선택만 짧게 적는다.
- `이유`에는 선택한 근거를 적는다.
- `버린 대안`에는 검토했지만 채택하지 않은 선택지를 적는다.
- `재검토 조건`에는 이 결정을 다시 논의해야 하는 조건을 적는다.
- 단순 구현 기록이나 작업 완료 기록은 `decisions.md`에 남기지 않는다.

## 오래된 메모리 정리 규칙

- 현재 코드와 맞지 않는 메모리는 수정하거나 삭제한다.
- 더 이상 유효하지 않은 결정은 `decisions.md`에서 상태를 `superseded`로 바꾼다.
- 오래됐지만 참고 가치가 있는 내용은 `archived/`로 이동한다.
- 더 이상 적용되지 않는 규칙은 `conventions.md`에 남기지 않는다.
- `tasks.md`의 완료된 작업은 오래 보존하지 않는다.
- 메모리 파일은 현재와 다음 작업 판단에 필요한 정보만 유지한다.

## archived/ 이동 기준

`archived/`는 현재 판단에는 직접 쓰지 않지만, 과거 맥락으로 보존할 가치가 있는 내용을 둔다.

다음 경우에 `archived/`로 이동한다.

- 현재 코드나 프로젝트 방향과 맞지 않지만 참고 가치가 있는 경우
- `decisions.md`의 `superseded` 항목이 누적되어 현재 결정을 읽기 어려운 경우
- `tasks.md`의 보류 항목이 더 이상 실제 작업 큐가 아닌 경우
- 사용자가 보존을 원하지만 현재 판단에는 쓰면 안 되는 내용인 경우

이동 주체와 방식:

- archive 이동 작업은 AI가 수행한다.
- 사용자가 archive를 요청하면 AI가 이동한다.
- AI가 오래된 메모리를 발견하면 사용자에게 알리고 이동하거나, 명백히 운영 메모리를 방해하는 경우 직접 이동한다.
- `project-brief.md`, `conventions.md`, Tech Stack, 핵심 설계 원칙의 archive 이동은 사용자 확인 후 수행한다.
- 초기에는 별도 하위 디렉터리 구조를 강제하지 않는다. 필요해질 때만 연도별 또는 문서별로 나눈다.

## source code/git과 충돌 시 우선순위

정보가 충돌할 때의 우선순위는 다음과 같다.

1. 현재 source code
2. 사용자의 최신 명시 지시
3. git history와 commit diff
4. `.memory/`의 현재 메모리
5. 오래된 문서와 `archived/` 내용

원칙:

- source code가 메모리보다 우선한다.
- 테스트 결과는 세션 시작 기준 자료가 아니라 구현/검증 단계에서 확인한 경우에만 판단 근거로 사용한다.
- git history는 변경 이력 확인에 사용하고, 현재 작업 진입점은 `current-state.md`를 사용한다.
- 메모리가 실제 코드와 충돌하면 메모리를 그대로 믿지 않는다.
- 충돌이 작업 결과에 영향을 주면 사용자에게 알리고, 필요한 경우 메모리를 갱신한다.
